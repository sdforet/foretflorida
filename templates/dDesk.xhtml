PHPTAL

PHPTAL is a templating engine for PHP. It has many benefits such as: separation of logic and presentation, a robust macro system, no speed loss, fails with helpful warnings for the developer, and forces production of clean XHTML. These and many other advangates of PHPTAL are listed on the group's site: PHPTAL.org. 

@file samplePage.php

/**
 * the "new" object will identify
 * an inner template passed as an
 * argument, which is inserted into
 * a parent template identified by
 * the class
 */
require_once "php/Decorator.php";
$page = new Decorator('myPage.xhtml');
/**
 * any buisness logic will reside
 * here and get passed to the
 * page object
 */
echo $page->execute();

Symfony

Symfony.com describes itself like this: Symfony is a set of PHP Components, a Web Application framework, a Philosophy, and a Community - all working together in harmony. No doubt this is a large community with support wordwide. I have found Symfony to be very useful with larger scale projects, but too robust on smaller ones. Symfony's greatest asset is the integration of its features, and its greatest weakness; the sharp learning curve to get started with the framework. Included below is a portion of "The Form Component" which builds a two element form (a numerical input and a button) Direct Injection, services.yml, TWIG, db Entity, and vaildation.yml, not included. This should give you a good idea about Symfony's learning curve. 

@file MyNewFormType.php

/**
 * 1 - Create a form class type in /myBundle/Forms
 */
namespace path\to\MyBundle\Form;
use Symfony\Component\Form\AbstractType;
use Symfony\Component\Form\FormBuilderInterface;
use Symfony\Component\OptionsResolver\OptionsResolverInterface;

class MyNewFormType extends AbstractType
{
   public function buildForm(FormBuilderInterface $builder, array $options)
   {
       $builder
           ->add('key', 'text', array(
               'label' => 'Your ID NUmber: ',
               'required' => true,
               'attr' => array(
                   'pattern' => '[0-9]*',
                   'inputmode' => 'numeric'
               )
           )
           ->add('continue', 'submit', array(
               'attr' => array(
                   'class' => 'btn btn-default'
               )
           ))
       ;
   }
   public function getName()
   {
       return 'myNewForm';
   }
   public function setDefaultOptions(OptionsResolverInterface $resolver)
   {
       $resolver->setDefaults([
            'data_class' => 'Path\To\Doctrine\Entity\MyDataEntity'
       ]);
   }
}

TWIG

TWIG is the self-described "flexible, fast, secure, template engine for PHP". I am sure that this is a valid description given it's wide acceptance in the Symfony community. I have found that twig is overly verbose, and not as designer/developer friendly as others such as PHPTAL. The main benefit (as viewed by others) to using twig is an extensive tag/filter/function/test library that it includes. According to twig, it violates separation of resposibilities to use strtoupper() in PHP so they provide their syntax filter {{ 'welcome'|upper }} to use in the template. Ok, that is nice but I don't agree entirely with the premise. They provide a good number of these to alter the information passed to the template, some of which border on buisness logic in the template. Of course you don't have to use many of these, and other template engines also include many logical filters in their engines.

Using Templates

  1. Templates must be named with double extension ".html.twig"
  2. The base template is evaluated first
  3. Child templates can overwrite the parent's blocks with content of their own (only one level)
  4. If it is not in a block, it will not be replaced
  5. Child templates must declare what they're overwriting by using the extends tag
  6. Variables are not scoped to included templates such as macros that are not in the same file (bad)

Three types of syntax in Twig:

  1. {% ... %} "Do Something": executes statements like conditional logic and for loops
  2. {{ ... }} "Say Something": print this route of variable or expression; filters use pipe
  3. {# ... #} "Comment": results of a comment aren't displayed
@file sample.html.twig

{# loop through the array with values #}
{% for info in array %}
   <h3>{{ info.name }}</h3>
   <p>{{ info.description }}</p>
   {# loop through the innerArray for this info #}
   {% for deepInfo in info.items %}
      {{ deepInfo.name }}
      {% if deepInfo.active == true %}
         <span>Show this text!</span>
      {% endif %}
      <p>{{ deepInfo.description|raw }}</p>
   {% endfor %}
   <hr />
{% endfor %}
